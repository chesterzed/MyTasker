# Миграции БД

Схема базы описывается **только** этими файлами. `init_db.py` при старте
накатывает недостающие миграции через `db/migrate.py`, отслеживая версию во
встроенном `PRAGMA user_version`.

## Как добавить миграцию

1. Создай файл `00N_краткое_описание.sql`, где `N` — на 1 больше последнего.
2. Впиши SQL одного логического изменения.
3. Всё. При следующем `python db/init_db.py` (или запуске бота) миграция
   применится, а `user_version` поднимется до `N`.

**Правила:**
- Одна миграция = одно логическое изменение.
- Уже существующие миграции **никогда не редактируются** задним числом —
  только новый файл. Иначе базы, где старая версия уже накатана, разойдутся.
- Номер строго на 1 больше предыдущего.

## Два уровня сложности в SQLite

| Изменение | Рецепт |
|---|---|
| **Добавить** колонку / таблицу / индекс | Легко, одной строкой: `ALTER TABLE users ADD COLUMN foo TEXT DEFAULT 'x';` |
| **Сменить дефолт / удалить колонку / изменить CHECK** | SQLite не умеет `ALTER COLUMN` → нужна пересборка таблицы (см. ниже) |

### Пересборка таблицы (для «тяжёлых» изменений)

```sql
-- пример: изменить дефолт/ограничение колонки в users
PRAGMA foreign_keys = OFF;

CREATE TABLE users_new (
    ...новое определение таблицы целиком...
);
INSERT INTO users_new (id, telegram_id, ...)
    SELECT id, telegram_id, ... FROM users;
DROP TABLE users;
ALTER TABLE users_new RENAME TO users;

-- пересоздать индексы и триггеры, которые были на старой таблице
CREATE INDEX IF NOT EXISTS ...;

PRAGMA foreign_keys = ON;
```

Существующим строкам новый дефолт сам не проставится — если нужно, добавь
`UPDATE users SET foo = 'x' WHERE foo IS NULL;` в ту же миграцию.
